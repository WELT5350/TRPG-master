## 1. 动机 / 用户故事

跑团爱好者面临一个共同困境：**想玩，但没人愿意当主持人（KP）**。CoC 跑团对 KP 要求极高——需熟记数百页规则书、即兴编织恐怖叙事、公正裁决检定、管理多个角色的 SAN 值与私密线索。这导致大量 3-5 人小团体"有玩家、无 KP"，跑团计划一再搁置。

一句话提炼：**用 AI 替代真人 KP，降低跑团门槛，让"有人就能跑"。**

---

## 2. 目标用户

想跑团但找不到真人 KP 的新手玩家，或无人愿当主持人的跑团爱好者。3-5 人小团体，有社交需求，可能线下聚在一起也可能远程连线。

---

## 3. 现有做法及其不足

| 现有做法 | 不足 |
|----------|------|
| 真人 KP 主持 | KP 稀缺，学习成本高（数百页规则书），主持一场 3-4 小时精力消耗极大 |
| 文字跑团（QQ 群 / Discord + 骰子机器人） | 无结构化规则判定，SAN、战斗等机制靠人工记忆；无私密信息分发；叙事质量依赖 KP 个人能力 |
| 跑团模拟器类 App（如 TRPG Engine） | 偏工具属性，提供骰子、地图、角色卡，但**没有 AI 叙事能力**，仍需要真人 KP |
| AI 聊天跑团（如直接跟 ChatGPT 跑） | 单人体验，不支持多人房间、角色分离、私密线索；AI 容易忘记规则、混淆检定等级 |
| 通用 VTT（Foundry VTT / Roll20） | 功能强大但配置复杂，面向硬核玩家，手机端体验差，无 AI 主持 |

**核心空白**：市场上不存在"手机端 + AI 主持 + 多人房间 + 规则硬编码 + 私密信息分发"的完整产品。

## 4. 本期范围与明确不做

### MS1 范围（第 1-2 周）

MS1 的目标是**"方向成立、骨架串联"**，不是实现全部功能。接口是真的，实现可为桩。

| # | 交付物 | 说明 |
|---|--------|------|
| 1 | 产品 Proposal 定稿 | 本文档，团队评审通过后 |
| 2 | 架构骨架入库 | 模块拆分、接口定义、数据模型——代码入库、能编译、调用关系成立 |
| 3 | 前端可交互原型 | 核心屏幕串联（登录→房间→叙事→对话→检定），可用 mock 数据驱动 |

核心屏幕流程：
```
加入房间 → 选择规则（CoC）→ 导入预设模组 → 背景介绍 → 角色卡创建
→ 等候游戏开启 → AI 主持 → 各玩家输入行动 → AI 推动剧情
→ 最终完成剧情 → AI 给出复盘
```

### 明确不做（本期）

- D&D / 狼人杀 / 血染钟楼 / 剧本杀（仅 CoC 7th）
- 纯粹平台类产品 / 纯粹角色扮演助手产品
- 付费与商城
- 语音输入（文字优先，语音不阻塞核心流程）
- AIGC 图片生成（场景插图延后到 Should）
- AI NPC / AI 补位玩家（延后到 Could）
- 赛后复盘（MS1 不实现，v1 作为 Should）
- 上传自定义模组（MS1 使用预设模组）

> v1 后续功能（复盘、AIGC 插图、语音输入、AI 补位玩家等）将另开独立 Proposal 设计，本文仅聚焦 MS1。
>
> 技术选型（前端框架、通信协议、AI 模型具体供应商选定等）属于架构设计范畴，将在 Proposal 定稿后由架构设计阶段独立产出，不在本 Proposal 范围内。

## 5. 关键决策与依据

### 决策 1：规则引擎硬编码 vs AI 裁决

| 方案 | 描述 | 优劣 |
|------|------|------|
| **A. 硬编码（选中）** | D100 检定、SAN 变化、战斗结算由后端代码执行 | ✅ 结果确定、可复现、零延迟 ❌ 需完整实现 CoC 7th 规则 |
| B. AI 裁决 | 将规则判定也交给 AI，AI 自行判断投骰结果等级 | ✅ 开发快 ❌ AI 易混淆检定等级（如要求投 D20）、同一次投骰不同次询问结果不一致、幻觉风险 |

**选择 A 的依据**：CoC 7th 规则是确定性计算，不应引入概率性系统。用户对"公平"的感知建立在规则可验证之上——若检定结果可被质疑"AI 偏袒"，信任即刻瓦解。AI 的强项是叙事，规则判定不是。

> 竞品参照：D&D Beyond 的骰子工具由客户端硬编码；Foundry VTT 的规则模块（如 CoC 7th game system）由 JavaScript 实现，不走远程判断。

### 决策 2：AI 叙事质量如何保证

| 方案 | 描述 | 优劣 |
|------|------|------|
| **A. 云端大模型（选中）** | 调用 API（具体模型待实测选定） | ✅ 叙事质量高、迭代快 ❌ 有 API 成本、依赖网络 |
| B. 本地开源模型 | 自部署，离线可用 | ✅ 零 API 成本 ❌ 中文 CoC 恐怖叙事质量未经验证，部署运维成本高 |
| C. 固定剧本（无 AI） | 预设分支文本，无动态生成 | ✅ 零成本、确定性高 ❌ 无法应对玩家自由输入，失去"AI 主持"的核心价值 |

**选择 A 的依据**：AI 叙事是本产品的核心体验——如果 AI 无法产出合格的克苏鲁恐怖叙事，产品不成立。具体选哪个模型（DeepSeek / Claude 等）属于技术实现决策，将在架构设计阶段通过实测（叙事质量 + 成本 + 延迟）选定。产品层面只确定方向：用云端大模型，不锁定具体供应商。

> 竞品参照：AI Dungeon 使用 OpenAI 系模型通过 prompt 工程约束叙事风格；Character.AI 自研模型专注角色一致性。我们的需求介于两者之间——开放叙事 + NPC 角色一致性。

### 决策 3：AI 返回结构化 JSON vs 自由文本

| 方案 | 描述 | 优劣 |
|------|------|------|
| **A. 结构化 JSON（选中）** | AI 返回 `{narration, skill_check, san_change, private_info, suggestions}` | ✅ 叙事与规则解耦、后端可校验和拦截非法检定 ❌ 需要 prompt 工程约束 AI 输出格式 |
| B. 自由文本 + 后解析 | AI 返回自然语言，后端用规则/正则提取检定指令 | ✅ prompt 简单 ❌ 解析脆弱、边界 case 多、易遗漏检定 |

**选择 A 的依据**：规则引擎硬编码的前提是"检定请求由后端发起，AI 仅建议是否检定以及目标值"。结构化 JSON 是这个解耦的接口契约。若 AI 自由发挥，可能出现"AI 直接宣布结果"或"AI 自创不存在的检定规则"，破坏规则引擎的权威性。

## 6. 基本概念与信息结构

### 领域核心概念

- **房间（Room）**：一次跑团会话的容器，包含 1 个 KP（AI）+ N 个玩家
- **角色卡（Character Sheet）**：玩家在游戏中的化身，包含属性、技能、SAN、HP、物品
- **模组（Module）**：预设剧本，包含场景图、NPC、物品、事件触发器、结局条件
- **场景（Scene）**：模组中的一个地点节点，包含描述、出口、在场 NPC、可发现物品
- **检定（Skill Check）**：由 AI 叙事触发、规则引擎执行的 D100 判定
- **私密信息（Private Info）**：仅特定玩家可见的线索，由 AI 生成、定向推送给目标玩家

### 角色卡数据模型

```json
{
  "id": "p1",
  "name": "杰克·布朗",
  "occupation": "私家侦探",
  "age": 32,
  "str": 50, "con": 60, "siz": 55, "dex": 65,
  "app": 60, "int": 70, "pow": 65, "edu": 70,
  "hp": 11, "maxHp": 11,
  "san": 65, "maxSan": 99,
  "luck": 55,
  "mp": 14, "maxMp": 14,
  "skills": {
    "spot_hidden": 65, "listen": 50,
    "library_use": 60, "persuade": 45,
    "fighting_brawl": 35, "firearms": 30,
    "stealth": 40, "first_aid": 40,
    "occult": 25, "psychology": 30
  },
  "inventory": ["手电筒", "笔记本", "左轮手枪(.38)", "急救包"]
}
```

### 场景数据模型（以"追书人"模组为例）

## 7. 规则引擎详述

CoC 7th 规则由硬编码执行，AI 仅负责叙事。检定判定不走 AI——这是本产品最核心的设计决策之一（依据见第 5 节决策 1）。

### D100 检定

| 投骰结果 | 等级 | 含义 |
|---------|------|------|
| 01 | 大成功 | 完美，额外正面效果 |
| ≤ 技能/5 | 极限成功 | 非常出色 |
| ≤ 技能/2 | 困难成功 | 高质量完成 |
| ≤ 技能值 | 常规成功 | 完成 |
| > 技能值 | 失败 | 未达成 |
| 96-100（技能<50） | 大失败 | 灾难性后果 |

### SAN 机制

- 起始 SAN = POW
- 遭遇神话事件损失 SAN（1/1D3 至 1D10/1D100）
- 单次损失 ≥5 → 暂时性疯狂
- SAN = 0 → 永久疯狂，角色退出

### 衍生属性

- HP = (CON + SIZ) / 10
- MP = POW / 5
- LUCK = 3D6 × 5

### 规则判定流程

```
AI 叙事 → 返回 JSON（含 skill_check 建议）
  → 后端规则引擎读取 skill_check.target + skill_check.skill
  → 玩家投骰（前端触发随机数，结果传给后端）
  → 后端按 D100 表判定等级
  → 结果广播至全员（公共）+ 触发 SAN 变化（如有）
```

> 此流程确保 AI 只建议"是否需要检定、目标值是多少"，实际判定永远由硬编码执行。即 AI 说"请投侦察"，但"投出 45 是常规成功还是困难成功"由代码决定——AI 无法越权宣布结果。

## 8. AI 交互设计

每次请求发给 AI 的上下文：
- 玩家输入
- 当前场景（地点、时间、在场 NPC）
- 角色状态（HP / SAN / 持有物品）
- 最近 6 轮对话历史
- 已获取的线索摘要

AI 返回结构化 JSON：

```json
{
  "narration": "铁门在生锈的铰链上发出一声刺耳的呻吟…",
  "skill_check": {
    "triggered": true,
    "skill": "spot_hidden",
    "skill_name": "侦察",
    "target": 65,
    "reason": "你注意到脚印的排列有些奇怪"
  },
  "san_change": null,
  "private_info": [
    {"player": "p1", "content": "你在门框缝隙发现了一片灰色布料"}
  ],
  "suggestions": ["检查脚印", "进入会客室", "上楼"]
}
```

叙事由 AI 生成，`skill_check` 由后端规则引擎硬编码判定。两者解耦。

### 端到端示例

以下用一个具体例子说明一轮交互的完整链路——这是规范本身，不是插图：

```
[当前场景] 惠特利旧宅 · 门厅。玩家A（杰克·布朗）在场。
[玩家A输入] "我蹲下来仔细检查门框下方的缝隙"

→ AI 收到上下文（场景描述 + 杰克的状态 HP11/SAN65 + 最近 6 轮对话 + 已有线索）
→ AI 返回 JSON：
   {
     "narration": "你单膝跪地，手指沿着门框的缝隙缓缓摸索。灰尘黏在指尖上，
                   就在你觉得一无所获时，指甲碰到了一片柔软的织物——",
     "skill_check": {
       "triggered": true,
       "skill": "spot_hidden",
       "skill_name": "侦察",
       "target": 65,
       "reason": "你需要仔细辨认这片织物的来源"
     },
     "san_change": null,
     "private_info": [
       {"player": "p1", "content": "这片灰色布料上沾有暗红色的痕迹——很可能是干涸的血。"}
     ],
     "suggestions": ["仔细检查布料", "查看周围是否有血迹", "叫其他玩家来看"]
   }

→ 后端规则引擎：skill_check.triggered = true
  → 通知玩家A投侦察（目标值 65）
→ 公共广播 narration 到全员："你单膝跪地，手指沿着门框..."
→ 私密信息仅推给玩家A："这片灰色布料上沾有暗红色的痕迹..."
→ 前端显示三个建议行动按钮

→ 玩家A 点击"投骰"
→ 前端触发 D100 随机数 → 结果为 23
→ 后端判定：23 ≤ 65（技能值），且 23 ≤ 65/2（困难成功阈值 32.5）
→ 等级：困难成功
→ 全员广播："侦察检定 — 困难成功！（23 vs 65）"
→ 后端生成额外信息，补充到 narration

→ 如果投出 97（技能 65 ≥ 50，不触发大失败）：
→ 等级：失败
→ 全员广播："侦察检定 — 失败。（97 vs 65）"
→ AI 收到失败结果，生成新的 narration："你翻遍了缝隙，除了灰尘什么也没找到。"
```

**此例同时覆盖边界情况**：
- 正常检定 + 成功（23 vs 65 → 困难成功）
- 正常检定 + 失败（97 vs 65 → 失败）
- 私密信息仅玩家A可见，公共叙事全员可见
- SAN 无变化时不触发 SAN 机制
- 投骰由前端触发，判定由后端硬编码执行

## 10. MS1 验收标准

- [ ] 打开 App → 创建/加入房间 → 开场叙事，流程无阻断
- [ ] 至少 3 轮"输入 → AI 回应 → D100 检定"完整循环
- [ ] AI 不犯规则错误（如要求投 D20、混淆检定等级、越权宣布结果）
- [ ] 前端核心屏幕串联，无阻断性 bug
- [ ] AI 返回的 JSON 被后端正确解析，叙事与检定解耦验证通过
