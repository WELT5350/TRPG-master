## 1. 关键决策与依据

> 以下三个决策各对应一份 MiniSpec 窄提案。本节仅做概要，完整备选方案分析 + 验收标准见对应文件。

### 决策 1：规则引擎硬编码 vs AI 裁决

| 方案 | 描述 | 优劣 |
|------|------|------|
| **A. 硬编码（选中）** | D100 检定、SAN 变化、战斗结算由后端代码执行 | ✅ 结果确定、可复现、零延迟 ❌ 需完整实现 CoC 7th 规则 |
| B. AI 裁决 | 将规则判定也交给 AI | ✅ 开发快 ❌ 易混淆检定等级、同次投骰结果不一致、幻觉风险 |

**选择 A 的依据**：CoC 7th 规则是确定性计算，不应引入概率性系统。用户对"公平"的感知建立在规则可验证之上。AI 的强项是叙事，规则判定不是。

> 竞品参照：D&D Beyond 骰子工具由客户端硬编码；Foundry VTT 的 CoC 7th game system 由 JavaScript 实现，不走远程判断。

详见：`docs/proposals/proposal-01-规则引擎.md`

### 决策 2：AI 叙事质量如何保证

| 方案 | 描述 | 优劣 |
|------|------|------|
| **A. 云端大模型（选中）** | 调用 API（具体模型待实测选定） | ✅ 叙事质量高、迭代快 ❌ 有 API 成本、依赖网络 |
| B. 本地开源模型 | 自部署，离线可用 | ✅ 零 API 成本 ❌ 中文 CoC 恐怖叙事质量未验证 |
| C. 固定剧本（无 AI） | 预设分支文本 | ✅ 零成本、确定性高 ❌ 无法应对自由输入，失去核心价值 |

**选择 A 的依据**：AI 叙事是核心体验——如果 AI 无法产出合格的克苏鲁恐怖叙事，产品不成立。具体模型（DeepSeek / Claude 等）将在架构设计阶段通过实测选定。

> 竞品参照：AI Dungeon 使用 OpenAI 系模型约束叙事风格；Character.AI 自研模型专注角色一致性。我们的需求介于两者之间。

详见：`docs/proposals/proposal-03-AI叙事方案.md`

### 决策 3：AI 返回结构化 JSON vs 自由文本

| 方案 | 描述 | 优劣 |
|------|------|------|
| **A. 结构化 JSON（选中）** | AI 返回 `{narration, skill_check, san_change, private_info, suggestions}` | ✅ 叙事与规则解耦、后端可拦截非法检定 ❌ prompt 复杂度高 |
| B. 自由文本 + 后解析 | AI 返回自然语言，后端用正则提取 | ✅ prompt 简单 ❌ 解析脆弱、边界 case 多 |

**选择 A 的依据**：规则引擎硬编码的前提是"检定请求由后端发起，AI 仅建议"。结构化 JSON 是这个解耦的接口契约。若 AI 自由发挥，可能出现"AI 直接宣布结果"破坏规则引擎权威性。

详见：`docs/proposals/proposal-02-AI输出格式.md`

## 2. 核心概念与信息结构

### 领域核心概念

- **房间（Room）**：一次跑团会话的容器，包含 1 个 KP（AI）+ N 个玩家
- **角色卡（Character Sheet）**：玩家在游戏中的化身，包含属性、技能、SAN、HP、物品
- **模组（Module）**：预设剧本，包含场景图、NPC、物品、事件触发器、结局条件
- **场景（Scene）**：模组中的一个地点节点，包含描述、出口、在场 NPC、可发现物品
- **检定（Skill Check）**：由 AI 叙事触发、规则引擎执行的 D100 判定
- **私密信息（Private Info）**：仅特定玩家可见的线索，由 AI 生成、定向推送

### 角色卡数据模型

```json
{
  "id": "p1",
  "name": "杰克·布朗",
  "occupation": "私家侦探",
  "age": 32,
  "str": 50, "con": 60, "siz": 55, "dex": 65,
  "app": 60, "int": 70, "pow": 65, "edu": 70,
  "hp": 11, "maxHp": 11,
  "san": 65, "maxSan": 99,
  "luck": 55,
  "mp": 13, "maxMp": 13,
  "skills": {
    "spot_hidden": 65, "listen": 50,
```

## 3. 规则引擎详述

CoC 7th 规则由硬编码执行，AI 仅负责叙事。检定判定不走 AI——这是本产品最核心的设计决策之一。

### D100 检定

| 投骰结果 | 等级 | 含义 |
|---------|------|------|
| 01 | 大成功 | 完美，额外正面效果 |
| ≤ 技能/5 | 极限成功 | 非常出色 |
| ≤ 技能/2 | 困难成功 | 高质量完成 |
| ≤ 技能值 | 常规成功 | 完成 |
| > 技能值 | 失败 | 未达成 |
| 96-100（技能<50） | 大失败 | 灾难性后果 |

> 技能≥50 时，仅投出 100 为大失败。

### SAN 机制

- 起始 SAN = POW
- 遭遇神话事件损失 SAN（1/1D3 至 1D10/1D100）
- 单次损失 ≥5 → 暂时性疯狂
- SAN = 0 → 永久疯狂，角色退出

### 衍生属性

- HP = (CON + SIZ) / 10
- MP = POW / 5
- LUCK = 3D6 × 5

### 规则判定流程

```
AI 叙事 → 返回 JSON（含 skill_check 建议）
  → 后端规则引擎读取 skill_check.target + skill_check.skill
  → 玩家投骰（前端触发随机数，结果传给后端）
  → 后端按 D100 表判定等级
  → 结果广播至全员（公共）+ 触发 SAN 变化（如有）
```

> AI 说"请投侦察"，但"投出 45 是什么等级"由代码决定——AI 无法越权宣布结果。

## 4. AI 交互设计

### 上下文构造

每次请求发给 AI：
- 玩家输入
- 当前场景（地点、时间、在场 NPC）
- 角色状态（HP / SAN / 持有物品）
- 最近 6 轮对话历史
- 已获取的线索摘要

### AI 响应格式

```json
{
  "narration": "铁门在生锈的铰链上发出一声刺耳的呻吟…",
  "skill_check": {
    "triggered": true,
    "skill": "spot_hidden",
    "skill_name": "侦察",
    "target": 65,
    "reason": "你注意到脚印的排列有些奇怪"
  },
  "san_change": null,
  "private_info": [
    {"player": "p1", "content": "你在门框缝隙发现了一片灰色布料"}
  ],
  "suggestions": ["检查脚印", "进入会客室", "上楼"]
}
```

叙事由 AI 生成，`skill_check` 由后端规则引擎硬编码判定。两者解耦。

### 端到端示例

以下用一个具体例子说明一轮交互的完整链路——这是规范本身，不是插图：

```
[当前场景] 惠特利旧宅 · 门厅。玩家A（杰克·布朗）在场。
[玩家A输入] "我蹲下来仔细检查门框下方的缝隙"

→ AI 收到上下文（场景描述 + 杰克的状态 HP11/SAN65 + 最近 6 轮对话 + 已有线索）
→ AI 返回 JSON：
   {
     "narration": "你单膝跪地，手指沿着门框的缝隙缓缓摸索。灰尘黏在指尖上，
                   就在你觉得一无所获时，指甲碰到了一片柔软的织物——",
     "skill_check": {
       "triggered": true,
       "skill": "spot_hidden",
       "skill_name": "侦察",
       "target": 65,
       "reason": "你需要仔细辨认这片织物的来源"
     },
     "san_change": null,
     "private_info": [
       {"player": "p1", "content": "这片灰色布料上沾有暗红色的痕迹——很可能是干涸的血。"}
     ],
     "suggestions": ["仔细检查布料", "查看周围是否有血迹", "叫其他玩家来看"]
   }

→ 后端规则引擎：skill_check.triggered = true
  → 通知玩家A投侦察（目标值 65）
→ 公共广播 narration 到全员："你单膝跪地，手指沿着门框..."
→ 私密信息仅推给玩家A："这片灰色布料上沾有暗红色的痕迹..."
→ 前端显示三个建议行动按钮

→ 玩家A 点击"投骰"
→ 前端触发 D100 随机数 → 结果为 23
→ 后端判定：23 ≤ 65（技能值），且 23 ≤ 65/2（困难成功阈值 32.5）
→ 等级：困难成功
→ 全员广播："侦察检定 — 困难成功！（23 vs 65）"
→ 后端生成额外信息，补充到 narration

→ 如果投出 97（技能 65 ≥ 50，不触发大失败）：
→ 等级：失败
→ 全员广播："侦察检定 — 失败。（97 vs 65）"
→ AI 收到失败结果，生成新的 narration："你翻遍了缝隙，除了灰尘什么也没找到。"
```

**此例同时覆盖边界情况**：
- 正常检定 + 成功（23 vs 65 → 困难成功）
- 正常检定 + 失败（97 vs 65 → 失败）
- 私密信息仅玩家A可见，公共叙事全员可见
- SAN 无变化时不触发 SAN 机制
- 投骰由前端触发，判定由后端硬编码执行

## 5. 原型

前端原型 `trpg-app`（凌铭辉 LMH 开发）已完成所有核心屏幕的交互串联，Mock 模式下无需后端即可运行。

### 技术栈

React 19 · TypeScript · Vite 8 · Tailwind CSS 3 · Zustand · React Router 7

### 运行

```bash
cd trpg-app && npm install && npm run dev   # → http://localhost:9877
```

### 已实现页面（10 屏）

| 页面 | 路径 | 功能 |
|------|------|------|
| 登录 | `/login` | 品牌页，扫码加入/创建房间/浏览游戏入口 |
| 加入房间 | `/join` | 输入 1-4 位房间码 |
| 创建房间 | `/create` | 设置房间名/房间码/人数上限，选择游戏和模组 |
| 游戏选择 | `/games` | 4 种游戏类型卡片（仅 TRPG 可用） |
| 系统选择 | `/games/:gameId` | CoC 7th（就绪）/ D&D 5e（开发中） |
| 模组选择 | `/games/:gameId/scenarios/:systemId` | 4 个 CoC 预设模组 + 2 个 D&D 模组 |
| 故事介绍 | `/story` | 模组背景叙事，1920 年代档案风格 |
| 角色创建 | `/character` | 4 步向导：信息→属性→技能→完成 |
| 大厅 | `/lobby` | 房间码展示、玩家槽位、准备/开始 |
| 游戏房间 | `/room` | 聊天 + 快捷操作 + D100/D20/D6 掷骰 + 底部 4 面板（角色卡/技能/地图/速记） |

### 核心亮点

- **拖拽掷骰系统**：鼠标/触屏拖拽产生随机种子，D100 显示十位/个位双骰，含成功等级判定
- **4 步角色创建向导**：职业搜索/分组筛选 → 8 属性加点 → 职业技能/兴趣技能分池分配 → 装备/背景/笔记
- **1920 年代档案美学**：羊皮纸色系、黄铜色点缀、打字机字体风格
- **手机优先**：最大 390px 宽度，PWA 支持可安装到桌面

## 6. MS1 验收标准

- [ ] 打开 App → 创建/加入房间 → 开场叙事，流程无阻断
- [ ] 至少 3 轮"输入 → AI 回应 → D100 检定"完整循环
- [ ] AI 不犯规则错误（如要求投 D20、混淆检定等级、越权宣布结果）
- [ ] 前端核心屏幕串联，无阻断性 bug
- [ ] AI 返回的 JSON 被后端正确解析，叙事与检定解耦验证通过
