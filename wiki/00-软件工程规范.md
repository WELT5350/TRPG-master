# XEngineer 营软件工程规范

## 核心原则

**先设计，再开发** — 需求未澄清前不写代码。

## 8 周 / 4 里程碑

| 里程碑 | 时间 | 目标 |
|--------|------|------|
| **MS0** | 开营前 | 预习与准备 |
| **MS1** | 第 1-2 周 | 战略决策 + 首尾可跑版本 |
| **MS2** | 第 3-4 周 | MVP 深化 |
| **MS3** | 第 5-6 周 | 功能闭环 + 质量 |
| **MS4** | 第 7-8 周 | 打磨、交付、发布 |

## MS1 交付物

1. 战略决策文档（产品方向与范围）
2. 产品提案（FullSpec）
3. 架构骨架（空壳代码，含真实接口定义）
4. 首尾可跑版本（贯通关键链路）

每个里程碑走完整的产品 → 架构 → 开发循环。

## 协作规则

- **Git / PR / Review 强制**：所有代码变更必须走 PR
- **Issue 驱动开发**：每个功能/任务对应一条 Issue
- **Fork + PR 工作流**：从 Fork 仓库向主仓库提 PR
- **文档先行**：代码前先出设计文档
- **每日站会**：当天完成 + 明天计划，有记录
- **周会**：导师反馈

## 文档哲学

- **工程文档**：存放在 Issue 中，定稿后不可变更，修改以新增而非覆写
- **用户文档**：存放在仓库中，持续维护更新

## AI 使用红线

- AI 可辅助，不可替代工程流程
- AI 生成的代码必须被人类理解
- 核心 PR 需附带使用的关键 Prompt
- 系统架构必须由人类设计

## 禁区（8 条）

1. 不写提案直接写代码
2. 绕过 PR / Review 流程
3. AI 生成无人理解的代码
4. 范围蠕动（scope creep）
5. 跳过测试
6. 大 PR（应小而频繁）
7. 文档过时不更新
8. 单点阻塞（不主动寻求帮助）
